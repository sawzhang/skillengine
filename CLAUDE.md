This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

SkillEngine is a framework-agnostic skills execution engine for LLM agents. It provides a plugin-based system for defining, loading, filtering, and executing skills (capabilities) that can be made available to large language models. Aligned with the Claude Agent Skills architecture (progressive disclosure, on-demand loading, per-skill model/tools).

**Package**: `skillengine` on PyPI | **Repo**: `sawzhang/skillengine`

## Versioning & Release

Version is derived automatically from git tags via `hatch-vcs`. **Never edit version strings manually.**

```bash
# Check current version
uv run python -c "import skillengine; print(skillengine.__version__)"

# Release workflow: tag ‚Üí push ‚Üí CI builds & publishes to PyPI
git tag v0.2.0
git push origin master --tags
```

**How it works**:
- `git tag v0.2.0` ‚Üí build produces `skillengine==0.2.0`
- Commits after tag ‚Üí `0.2.1.dev3` (auto dev version)
- CI pipeline: `lint` ‚Üí `test` (3.10/3.11/3.12) ‚Üí `build` ‚Üí `publish` (only on `v*` tags)
- Publishing uses PyPI trusted publishers (OIDC, no token needed in CI)

**Files involved**:
- `pyproject.toml`: `dynamic = ["version"]` + `[tool.hatch.version] source = "vcs"`
- `src/skillengine/_version.py`: auto-generated by hatch-vcs at build time (gitignored)
- `src/skillengine/__init__.py`: imports `__version__` from `_version.py`
- `.github/workflows/ci.yml`: publish job triggers on `refs/tags/v*`

### Core Components

- **AgentRunner** (`agent.py`): The main orchestrator. Manages conversation loop, on-demand skill loading via `skill` tool, `$ARGUMENTS` substitution, `context: fork` isolation, `!`cmd`` dynamic injection, per-skill model switching, per-skill tool restrictions, streaming, multi-modal messages, session persistence.
- **SkillsEngine** (`engine.py`): Pipeline coordinator. Loads skills from directories, applies filters, generates prompts (XML/Markdown/JSON), executes commands, supports file watching for hot-reload.
- **Models** (`models.py`): Core data structures ‚Äî `Skill`, `SkillMetadata`, `SkillRequirements`, `SkillSnapshot`, `SkillEntry`, `SkillAction`, `SkillInvocationPolicy`, `SkillInstallSpec`, multi-modal `MessageContent`.
- **Config** (`config.py`): `SkillsConfig` for directory management, filtering, per-skill overrides (`SkillEntryConfig`), `CacheRetention`.

### Plugin Subsystems

| Subsystem | Base Class | Implementations | Purpose |
|-----------|-----------|-----------------|---------|
| Loaders | `SkillLoader` | `MarkdownSkillLoader` | Parse skill files (Markdown + YAML frontmatter) |
| Filters | `SkillFilter` | `DefaultSkillFilter` | Check eligibility (bins, any_bins, env, os, config) |
| Runtimes | `SkillRuntime` | `BashRuntime`, `CodeModeRuntime`, `BoxLiteRuntime` | Execute with timeout, streaming, sandboxing |
| Adapters | `LLMAdapter` | `OpenAIAdapter`, `AnthropicAdapter` | Provider integration with tool calling |

### Events System

`EventBus` with 13 event types, priority-based handlers, and blockable results:

- **Lifecycle**: `AGENT_START`, `AGENT_END`, `SESSION_START`, `SESSION_END`
- **Turn**: `TURN_START`, `TURN_END`
- **Tool**: `BEFORE_TOOL_CALL` (blockable), `AFTER_TOOL_RESULT`, `TOOL_EXECUTION_UPDATE`
- **Context**: `CONTEXT_TRANSFORM`, `INPUT`, `COMPACTION`
- **Model**: `MODEL_CHANGE`

`BEFORE_TOOL_CALL` handlers can return `ToolCallEventResult(block=True)` to prevent execution.

### Skill Definition Format

Skills are Markdown files with YAML frontmatter at `skills/<name>/SKILL.md`:

```yaml
---
name: skill-name
description: What the skill does
model: claude-sonnet-4-20250514   # Per-skill model override
context: fork                      # Isolated subagent execution
allowed-tools: [Read, Grep]       # Tool restrictions
argument-hint: "<query>"           # Autocomplete hint
user-invocable: true               # Enable /skill-name slash command
disable-model-invocation: false    # Hide from LLM auto-discovery
hooks:
  PreToolExecution: "echo pre"
metadata:
  emoji: "üîß"
  primary_env: "API_KEY"
  requires:
    bins: ["git"]                  # ALL must exist
    any_bins: ["npm", "pnpm"]     # At least ONE must exist
    env: ["GITHUB_TOKEN"]
    os: ["darwin", "linux"]
  install:                         # Installation hints

```

### On-Demand Skill Loading

System prompt contains only skill **names and descriptions** (metadata). Full content loaded on-demand when LLM calls the `skill` tool. This follows the progressive disclosure pattern.

- `AgentConfig.skill_description_budget` (default 16000) limits total chars for skill metadata in system prompt
- `$ARGUMENTS`, `$1`..`$N`, `${CLAUDE_SESSION_ID}` substituted in skill content
- `!`command`` placeholders replaced with command stdout before sending
- `context: fork` ‚Üí isolated child `AgentRunner` with skill content as system prompt
- `model:` ‚Üí `switch_model()` before, restore in `finally` block
- `allowed-tools:` ‚Üí restrict tools during skill execution

### Skill Validation

`AgentRunner.validate_skill(skill)` enforces:
- Name: ‚â§64 chars, lowercase alphanumeric + hyphens, no leading hyphen
- Description: non-empty, ‚â§1024 chars

### CodeModeRuntime

Inspired by Cloudflare's code-mode-mcp. Exposes 2 tools (`search` + `execute`) instead of N tools. LLM writes Python code against injected data and clients. Token cost O(1).

```python
runtime = CodeModeRuntime(
    spec=openapi_spec,              # Any data for discovery
    ctx={"client": httpx.Client()}, # Objects injected into execute mode
)
await runtime.search("[p for p in spec['paths'] if '/users' in p]")
await runtime.run("result = ctx['client'].get('/users')")
tools = runtime.get_tool_definitions()  # ‚Üí [search, execute]
```

- `search(code)` injects `spec` only; `run(code)` / `execute(code)` injects both `spec` and `ctx`
- Sandbox modes: `"inprocess"` (restricted builtins) and `"subprocess"` (child process, JSON-serializable only)
- Safe modules: json, re, math, datetime, collections, itertools, functools, urllib.parse
- Result convention: assign to `result` for structured output, or `print()` for text

### Context Management

- `ContextManager` coordinates compaction strategies
- `TokenBudgetCompactor` ‚Äî keep messages within a token budget
- `SlidingWindowCompactor` ‚Äî keep N most recent messages
- `estimate_tokens()` ‚Äî chars/4 heuristic for token estimation

### Session Persistence

JSONL append-only tree structure (`session/`):
- `SessionManager`: create, load, append, branch, walk_to_root
- Entry types: message, model change, thinking level change, label, branch summary, compaction, custom
- Branching: fork at any entry point to create alternative conversation paths

### Model Registry

`ModelRegistry` with built-in catalog (`models_catalog.py`):
- Providers: Anthropic, OpenAI, Google, DeepSeek, MiniMax
- Per-model: context_window, max_output_tokens, cost, capabilities, reasoning support
- `ThinkingLevel`: off/minimal/low/medium/high/xhigh ‚Äî mapped to provider-specific budgets
- `TokenUsage` + `CostBreakdown` for cost tracking
- Prompt caching: `get_cache_control_anthropic()`, `get_cache_config_openai()`

### Extensions

`ExtensionManager` discovers plugins from:
- Python entry points (`skillengine.extensions`)
- `~/.skillengine/extensions/*.py`
- `./.skillengine/extensions/*.py`

Each extension receives `ExtensionAPI` with: `on()`, `register_command()`, `register_tool()`, `register_adapter()`.

### Execution Modes

- **Interactive** (`interactive.py`): Terminal REPL with TUI, autocomplete, markdown rendering
- **JSON** (`json_mode.py`): Single-shot, streams JSONL events to stdout
- **RPC** (`rpc_mode.py`): Persistent JSON-line protocol ‚Äî prompt, steer, follow_up, abort, get_state, set_model, fork, switch_session

### Key Patterns

- **Progressive Disclosure**: Only skill metadata in system prompt; full content on-demand via `skill` tool
- **Environment Management**: Context manager (`env_context`) safely backup/restore env vars for per-skill overrides
- **Snapshot Caching**: `SkillSnapshot` provides immutable views with version tracking and content hashing
- **Filter Short-circuiting**: Eligibility checks sequence and short-circuit on first failure
- **Per-Skill Model Switching**: `switch_model()` before skill content, restore in `finally` block
- **Fork Isolation**: `_execute_skill_forked()` creates child `AgentRunner` with skill content as system prompt
- **Adapter Registry**: `AdapterFactory` for lazy adapter instantiation, extensible via plugins
- **Event-Driven Control**: `BEFORE_TOOL_CALL` can block tool execution; `CONTEXT_TRANSFORM` can modify messages
